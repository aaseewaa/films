### Понедельник, 4 мая — Проектирование БД

Цель дня: спроектировать структуру базы данных под граф знаний о кинематографе.

Сделано:
- ER-диаграмма в **dbdiagram.io** с базовой структурой: film, person, director_influence, entity, entity_translation
- Принято ключевое архитектурное решение — **class table inheritance** через единую таблицу `entity` с дочерними `film`, `person`, `article`, `collection`. Все связи между разнородными сущностями (переводы, теги, медиа, эмбеддинги) идут через `entity_id`. Это даёт полиморфные связи и одну инфраструктуру для всех типов.

Решения для защиты:
- Class Table Inheritance — стандартный паттерн (Fowler, *Patterns of Enterprise Application Architecture*) для типизированной иерархии в реляционной БД. Альтернатива — Single Table или Concrete Table — отвергнуты: Single Table даёт разреженность колонок (у фильма не должно быть `is_director`), Concrete Table не позволяет полиморфных связей.

- Метрика — **cosine distance** (оператор `<=>` в pgvector). Соответствует обучению sentence-transformer моделей с `normalize_embeddings=True` и не зависит от длины вектора.

### Понедельник, 11 мая — Wikidata-граф влияний

Цель: загрузить связи «вдохновитель → ученик» для режиссёров через Wikidata.

Сделано:
- `scripts/wikidata_client.py` — SPARQL-клиент к Wikidata
- `scripts/load_wikidata_influences.py` — загрузчик связей через свойство **P737 "influenced by"**
- Реализована таблица `director_influence (source_director_id, target_director_id, weight, confidence, relation_note, inferred_by_system)` — направленные связи с метаданными источника
- Загружено **263 связи** для существующих в БД режиссёров

Трудности:
- Wikidata SPARQL имеет агрессивный rate limit (5 req/min для анонимных). Решено batch-режимом по 10 QID за запрос с паузой 8 секунд между батчами.
- Часть режиссёров из Wikidata не находится в БД (нет TMDB id). Решено через **fuzzy fallback**: TMDB find-by-imdb_id для попыток связать.

Решения для защиты:
- Wikidata выбрана как источник графа влияний потому что это единственная открытая БД с **community-edited связями** между киноперсонами. Альтернатива — экспертная разметка вручную, но это вне масштаба MVP.

### Среда, 13 мая — Граф API и content-based рекомендации

Цель: API для визуализации графа и базовый алгоритм рекомендаций.

Сделано:
- `/api/graph/director/{id}?depth=N` — подграф вокруг режиссёра через **рекурсивный CTE** в PostgreSQL
- `/api/graph/full?limit=50` — топ-N режиссёров с их связями для главной страницы
- Базовый content-based recommendations: похожие фильмы через пересечение жанров + actors
- `/api/recommendations?for_film_id=...&mode=content`

Трудности:
- Рекурсивный CTE с глубиной 3 от плотного хаба (Скорсезе) возвращал 50+ узлов — каша. Добавлен параметр `max_nodes`.
- asyncpg не выводит тип параметра в первой строке CTE. Лечение — явный `CAST(:center_id AS bigint)`.

Решения для защиты:
- Рекурсивный CTE даёт глубину 2 за 5-15 мс против 200-400 мс при итерации в Python (N round-trip). Это на порядок быстрее. `UNION` (без `ALL`) автоматически фильтрует посещённые узлы — встроенная защита от циклов.

### Четверг, 14 мая — TMDB keywords + pgvector + семантический поиск

Цель: подключить семантический поиск как главную фишку проекта.

Сделано:
- Догружены **TMDB keywords** (атмосферные теги вроде "memory", "time-travel", "isolation") — расширили `entity_taxonomy` новым типом `keyword`. Загружено ~1500-3000 keywords на 5000+ связях.
- Расширение pgvector установлено в БД, миграция `migration_pgvector.sql`
- Добавлена колонка `entity_translation.embedding vector(384)`
- `scripts/generate_embeddings.py` — генерация эмбеддингов через `sentence-transformers` (multilingual-MiniLM-L12-v2)
- Сгенерированы эмбеддинги для всех **10 596 переводов** (100% покрытие, RU + EN тексты)
- HNSW-индекс с `vector_cosine_ops` (`m=16, ef_construction=64`)
- `/api/search?mode=semantic` — семантический поиск
- `/api/recommendations?mode=semantic` — семантические рекомендации

Трудности:
- PyTorch требует numpy<2.0, иначе `_ARRAY_API not found`. Зафиксирован numpy==1.26.4.
- При массовой загрузке keywords первая ошибка отравляла asyncpg-сессию (`InFailedSQLTransactionError`). Решено per-record commit/rollback. Скорость упала с 500/сек до 50/сек, но загрузка устойчива.

Решения для защиты:
- HNSW (Hierarchical Navigable Small World) — современный алгоритм приближённого ближайшего соседа. Альтернатива в pgvector — IVFFlat, но требует обучения индекса и хуже на маленьких наборах данных. HNSW работает с пустой таблицей и масштабируется до миллионов векторов.
- Эмбеддинги на уровне `entity_translation` (а не `entity`) — потому что русский и английский тексты дают **разные векторы**. Поиск работает per-language.

- **Главная страница — радиальный граф влияний на чистом SVG**:
  - Случайный центр из топ-10 любимых режиссёров (Хичкок, Кубрик, Тарковский и др.)
  - 4 позиции на 360° (12/3/6/9 часов) — топ-4 учителя по weight
  - Серые placeholder-заглушки на пустых позициях для разреженных режиссёров
  - Клик на узел = новый центр + плавная перестройка графа

Итерации (важно для защиты):
- Первая попытка — `react-force-graph-2d` с физической симуляцией. Результат — облако точек без иерархии, нечитаемо.
- Вторая попытка — force-graph с центром и BFS на 3 уровня. Результат — каша из 56 узлов на хабах.
- **Финальная реализация — детерминированный SVG layout без физики**. 4 позиции рассчитаны математически (через `Math.cos/sin` от угла). Это даёт **гарантированно читаемую звезду** на любой БД, независимо от плотности связей.

Решения для защиты:
- Отказ от force-directed в пользу radial layout — сознательное решение исходя из принципов визуализации графов знаний. Радиальные layouts читаемы до глубины 2-3 (Munzner, *Visualization Analysis & Design*, MIT Press, 2014). Выбран **навигационный паттерн исследования** — пользователь видит ближайшее окружение и углубляется через клик. Альтернатива — обзорный режим со всеми связями — даёт «звёздное небо без созвездий».
- Топ-N по `weight` вместо всех связей — потому что в `director_influence` у популярного режиссёра может быть 20+ связей (Куарон, Мендес — хабы в графе). Без фильтрации по силе получается визуальный шум.

### Технические решения третьей недели

- **Backdrop хранится в `entity.primary_backdrop_url`** + дополнительно в `entity_media` с `role='backdrop' is_primary=true`. Симметрично существующему `primary_image_url`. Это удобный быстрый доступ без JOIN, при сохранении архитектурной целостности (галерея через `entity_media`).
- **Сортировка кадров по `vote_average`** из TMDB — community-edited метрика «голосования» пользователей TMDB за лучшие кадры. Лучший идёт как `primary_backdrop_url`, остальные — в галерею `stills` отсортированные по убыванию.
- **Идемпотентность всех скриптов** — каждый загрузчик можно запускать многократно без дублей. Это критично для разработки в одиночку: можно прервать прогон, исправить баг, перезапустить с того же места.

## Сквозные технические решения проекта

Эти решения важны для текстовой части диплома (раздел «Технические решения и обоснования»):

### Class Table Inheritance через entity
Базовый класс `entity` с дискриминатором `entity_type`. Наследники (film, person, article, collection) через FK `id REFERENCES entity(id) CASCADE`. Даёт единую инфраструктуру (переводы, теги, медиа, эмбеддинги) и полиморфные связи. Цена — дополнительный JOIN в каждом запросе.

### Многоязычность через отдельную таблицу
`entity_translation` со composite PK `(entity_id, language_id)`. Альтернатива (JSONB-поля) отвергнута — невозможен полнотекстовый поиск с языкозависимым стеммингом и невозможны векторные индексы.

### Семантический поиск
pgvector + multilingual MiniLM-L12-v2 (384 dims). Единое векторное пространство для русского и английского — русский запрос находит английские описания без перевода. HNSW-индекс с cosine distance.

### Дедупликация при импорте
`entity.external_ids` JSONB хранит идентификаторы всех источников (TMDB, Wikidata QID, IMDb). При загрузке скрипт сначала ищет по external_id, потом по fuzzy-name. Это позволяет связывать данные из разных источников.

### Граф влияний
Wikidata P737 (community-edited). 310 связей — это **потолок источника**, не БД. Архитектура поддерживает расширение через NLP-анализ биографий и экспертную разметку (для пост-MVP). Радиальная визуализация на фронте использует только топ-N связей по `weight` для читаемости.

### Per-record commit/rollback в загрузчиках
Из-за особенности asyncpg (`InFailedSQLTransactionError` при первой ошибке в транзакции) каждый INSERT в отдельной транзакции с try/except. Цена — производительность падает с ~500/сек до ~50/сек, но загрузка устойчива к любым ошибкам и может перезапускаться с места обрыва.
